# 家庭媒体本地服务启动指南

本文档说明如何在当前工作区启动家庭媒体本地开发环境。

适用目录：

- 工作区根目录：`/Users/qinwei/profile/家庭媒体`

## 启动后会有哪些服务

正常启动后，本地会有以下服务：

- MySQL：`127.0.0.1:3306`
- Backend：`http://127.0.0.1:8080/code`
- Admin Web：`http://127.0.0.1:5173/`
- MinIO API：`http://127.0.0.1:9000`
- MinIO Console：`http://127.0.0.1:9001`

## 一、启动前准备

### 1. 确认当前目录

先进入工作区根目录：

```bash
cd /Users/qinwei/profile/家庭媒体
```

### 2. 确认代码仓库已经就位

当前工作区使用根仓库 + 子仓库模式，至少要确保下面两个目录存在：

- `apps/backend`
- `apps/admin-web`

如果你是第一次在这台机器上准备工作区，可以执行：

```bash
./scripts/bootstrap-workspace.sh
```

它会按 `workspace.repos.json` 拉取后端和前端仓库。

### 3. 安装或确认本地依赖

启动脚本依赖以下本地环境：

- Java 17
- MySQL 命令行工具和服务能力
- 系统 `node`
- 系统 `npm`
- `python3`

其中：

- 后端启动依赖 Java 17
- 前端启动依赖系统 `node` 和 `npm`
- 整套启动脚本会尝试启动 MySQL，并自动创建 `db_code` 数据库
- MinIO 二进制使用 `infra/minio/minio`

### 4. 先执行一次环境自检

最推荐的第一步是：

```bash
./scripts/doctor.sh
```

这个脚本会检查：

- 目录是否存在
- `node`、`npm`、Java 17、`mysqladmin` 是否可用
- MinIO 二进制是否存在
- 端口是否已经被占用
- `.run/pids` 下是否有陈旧 pid 文件
- 日志文件是否存在

如果这里已经报错，先不要急着启动服务，优先按提示修复环境。

## 二、推荐启动方式：一键启动整套环境

如果你希望一次把数据库、对象存储、后端、前端都拉起来，直接执行：

```bash
./scripts/start-stack.sh
```

这个脚本会按顺序做下面几件事：

1. 检查 `apps/backend` 和 `apps/admin-web` 是否存在
2. 检查系统 `node` / `npm`
3. 检查 Java 17
4. 启动 MySQL
5. 确保数据库 `db_code` 存在
6. 启动 MinIO
7. 自动确保 MinIO 中存在 `media` bucket
8. 启动后端
9. 启动前端

启动成功后，你可以直接访问：

- 后端：`http://127.0.0.1:8080/code`
- 前端：`http://127.0.0.1:5173/`
- MinIO Console：`http://127.0.0.1:9001`

MinIO 默认账号：

- 用户名：`minioadmin`
- 密码：`minioadmin`

## 三、分步启动方式

如果你想单独排查某一层，建议按下面顺序逐步启动。

### 1. 启动 MinIO

```bash
./scripts/start-minio.sh
```

启动成功后可访问：

- API：`http://127.0.0.1:9000`
- Console：`http://127.0.0.1:9001`

### 2. 启动后端

```bash
./scripts/start-backend.sh
```

这个脚本会用 Gradle 启动 Spring Boot，并带上：

- `--spring.profiles.active=produce`

成功后访问：

- `http://127.0.0.1:8080/code`

### 3. 启动前端

```bash
./scripts/start-admin-web.sh
```

这个脚本本质上会从 `apps/admin-web` 里直接调用 Vite：

```bash
node node_modules/vite/bin/vite.js --host 0.0.0.0 --port 5173
```

成功后访问：

- `http://127.0.0.1:5173/`

## 四、如何确认服务是否真的起来了

### 1. 查看整体状态

```bash
./scripts/status-stack.sh
```

你会看到：

- MinIO
- MinIO Console
- MySQL
- Backend
- Frontend

分别是 `running` 还是 `stopped`。

### 2. 再做一次环境检查

```bash
./scripts/doctor.sh
```

如果服务已经正常启动，通常会看到对应端口处于监听状态。

### 3. 直接打开地址验证

建议至少检查下面三个地址：

- 前端首页：`http://127.0.0.1:5173/`
- 后端接口根路径：`http://127.0.0.1:8080/code`
- MinIO Console：`http://127.0.0.1:9001`

## 五、查看日志

如果服务启动失败，优先看日志。

查看后端日志：

```bash
./scripts/logs-backend.sh
```

查看前端日志：

```bash
./scripts/logs-admin-web.sh
```

MinIO 日志默认在：

- `.run/logs/minio.log`

MySQL 日志默认在：

- `.run/logs/mysql.log`

## 六、停止服务

停止整套环境：

```bash

```

分别停止单个服务：

```bash
./scripts/stop-minio.sh
./scripts/stop-backend.sh
./scripts/stop-admin-web.sh
```

如果你修改了脚本或环境，也可以直接重启整套服务：

```bash
./scripts/restart-stack.sh
```

## 七、常见问题和处理办法

### 1. `./scripts/doctor.sh` 提示找不到 `node` 或 `npm`

说明前端依赖的系统 Node.js 环境没有准备好。

先确认：

```bash
node -v
npm -v
```

如果命令不存在或执行失败，需要先修复本机 Node.js 环境。

### 2. `./scripts/start-backend.sh` 提示 Java 17 不可用

先确认：

```bash
/usr/libexec/java_home -v 17
```

如果没有输出，说明本机没有正确安装 Java 17。

### 3. MinIO 启动失败

如果你是 macOS，常见原因是 MinIO 二进制带有系统安全属性。

可以执行：

```bash
xattr -d com.apple.provenance ./infra/minio/minio
```

然后重试：

```bash
./scripts/start-minio.sh
```

### 4. 提示 pid 文件存在，但进程其实不在

这是典型的陈旧 pid 文件问题。

可以删除 pid 文件后重试：

```bash
rm -f ./.run/pids/minio.pid ./.run/pids/backend.pid ./.run/pids/frontend.pid ./.run/pids/mysql.pid
```

然后重新执行：

```bash
./scripts/start-stack.sh
```

### 5. 后端端口 `8080` 起不来

优先执行：

```bash
./scripts/logs-backend.sh
```

然后确认：

- Java 17 是否正常
- MySQL 是否已启动
- `db_code` 数据库是否可创建

### 6. 前端端口 `5173` 起不来

优先执行：

```bash
./scripts/logs-admin-web.sh
```

然后确认：

- `apps/admin-web/node_modules` 是否已安装
- 系统 `node` / `npm` 是否可执行

如果前端依赖还没装，可以进入前端目录执行：

```bash
cd apps/admin-web
npm install
```

装完后再回到根目录启动：

```bash
cd /Users/qinwei/profile/家庭媒体
./scripts/start-admin-web.sh
```

## 九、当前这台机器已发现的启动阻塞点

基于一次实际执行 `./scripts/doctor.sh` 的结果，当前机器至少有以下几个需要先处理的问题。

### 1. 系统 `node` 和 `npm` 已损坏

当前报错特征是：

```bash
dyld: Library not loaded: /opt/homebrew/opt/icu4c/lib/libicui18n.73.dylib
```

这通常说明 Homebrew 安装的 Node.js 依赖库已经不匹配，导致：

- `node -v` 无法正常执行
- `npm -v` 无法正常执行
- 前端 `./scripts/start-admin-web.sh` 无法启动

建议先修复：

```bash
brew reinstall icu4c node
node -v
npm -v
```

只有当 `node -v` 和 `npm -v` 都能正常输出版本号时，再继续启动前端。

### 2. MinIO 二进制还带有 macOS 安全属性

当前 `doctor.sh` 已检测到：

- `MinIO binary still has com.apple.provenance attribute`

建议执行：

```bash
xattr -d com.apple.provenance ./infra/minio/minio
```

然后重新尝试：

```bash
./scripts/start-minio.sh
```

### 3. `.run/pids` 下存在陈旧 pid 文件

当前已经发现以下服务存在“pid 文件还在，但进程不在”的情况：

- MinIO
- Backend
- Admin web

建议先清理：

```bash
rm -f ./.run/pids/minio.pid ./.run/pids/backend.pid ./.run/pids/frontend.pid ./.run/pids/mysql.pid
```

清理后再执行整套启动：

```bash
./scripts/start-stack.sh
```

### 4. 当前最推荐的修复后重试顺序

在这台机器上，建议按下面顺序处理：

```bash
cd /Users/qinwei/profile/家庭媒体
brew reinstall icu4c node
xattr -d com.apple.provenance ./infra/minio/minio
rm -f ./.run/pids/minio.pid ./.run/pids/backend.pid ./.run/pids/frontend.pid ./.run/pids/mysql.pid
./scripts/doctor.sh
./scripts/start-stack.sh
./scripts/status-stack.sh
```

### 5. 这次排查里哪些不是第一优先级问题

本次检查同时确认了下面这些事实：

- MySQL `3306` 当前已经在监听
- Java 17 已检测到
- `apps/backend`、`apps/admin-web`、`infra/minio`、`storage/minio-data` 目录都存在

所以当前最优先处理的不是数据库和 Java，而是：

1. 修好 `node` / `npm`
2. 清理 MinIO 安全属性
3. 清理陈旧 pid 文件
